    3.2 使用互斥量保护共享数据

    访问数据前，使用互斥量将相关数据锁住，当访问数据结束，再将数据解锁。但互斥量自身也有问题，也会造成死锁，或是对数据保护的太多或太少

    C++ 中使用互斥量

    • std::mutex 创建互斥量
    • 调用成员函数 lock() 进行上锁，成员函数 unlock() 进行解锁
      • 不推荐调用成员函数，因为这必须记住在每个函数出口都需要调用 unlock()

    • C++ 为互斥量提供了一个 RAII 语法的模板类 std::lock_guard

      • 其会在构造时提供已经上锁的互斥量，在析构时进行解锁

    • 举个例子

    • 大多数情况下，互斥量通常会与保护的数据放在同一个类中，而不是定义成全局变量，类似处理受保护数据的函数被定义为成员函数

    • 但是当一个成员函数返回的是保护数据的指针或引用时，会破坏对数据的保护
    • 具有访问能力的指针或引用可以访问并能修改被保护的数据，而不会被互斥锁限制

    发现结构内在的条件竞争

    即便你在类内部对共享数据做了互斥保护，如果接口将受保护的数据的引用或指针“泄露”给外部使用，锁就没能真正保护住那部分代码。

    • stack 容器的实现

      • 虽然 empty() 和 size() 在被调用并返回时是正确的，但其结果并不可靠；当它们返回后，其他线程就可以自由的访问栈，并可能 push() 多个元素到栈中，也可能 pop() 一些已经在栈中的元素。这样的话，empty() 和 size() 得到的结果就有问题了

    • 这种问题的解决办法就是改变接口设计

    • 选项1：传入一个引用
      • 第一个选项就是将变量的引用作为参数，传入 pop() 函数中获取想要的弹出值
      • 这种方法在大多数情况下不错，但是有个问题：需要临时构造一个堆中类型的实例，用于接收目标值
    • 选项2：无异常抛出的拷贝构造函数和移动构造函数
    • 选项3：返回指向弹出值的指针
      • 建议使用智能指针，
    • 选项4：“选项1+选项2”或“选项1+选项3”

    线程安全的堆栈类定义

    • 接口没有条件竞争，实现了选项 1 和 3 ，重载了pop()

      • 栈可以拷贝，拷贝构造函数对互斥量上锁，再进行拷贝
      • 当栈为空时，pop() 函数会抛出一个 empty_stack 异常，所以在 empty() 函数被调用后，其他部件还能正常工作
      • 使用 std::shared_ptr 避免内存分配管理问题
      • 堆栈中的五个操作，现在就剩下三个：push(), pop(), empty()。简化接口有利于数据控制，可以保证互斥量将一个操作完全锁住

    ThreadSafe_Stack 之所以线程安全，主要得益于几点设计

    • 所有对内部容器的访问都被互斥量保护，保证同一时间只有一个线程能读写底层 std::stack<T> data
    • 将互斥量声明为 mutable
      • 允许即便在 const 成员函数中也能合法的修改（加锁/解锁）互斥量
    • 异常安全与返回值策略
      • pop() 在弹出栈元素前首先测试 empty() 并抛出异常，且抛出异常前锁仍在作用域内，所以异常路径也能正常释放锁。
      • 在锁的作用域范围内构造返回值，然后再修改容器，保证了即便外部消费 shared_ptr<T> 时，栈内元素已经被移除，也不会出现悬空引用
    • 禁止赋值、拷贝构造安全
      • 赋值操作定义为 delete，避免不同实例间的不当数据竞争
      • 拷贝构造时对源对象加锁后再拷贝其 data，保证拷贝期间源栈不会被其他线程修改
    • 综上：
      • 互斥量保护：串行化对 data 的所有操作
      • RAII 加锁：借助 lock_guard，即使异常也能正常解锁
      • 智能指针返回值：避免裸引用/指针引发的悬空或数据冲突
      • 拷贝/赋值控制：保证类实例间不会无意中共享内部状态

    死锁：问题描述及解决方案

    一个给定操作需要两个或两个以上的互斥量时，就会产生死锁(deadlock) 问题

    • 避免死锁的一般建议：让两个互斥量总以相同的顺序上锁，即总是在互斥量 B 之前锁住互斥量 A，就永远不会死锁。但是即使在代码上做到这样，在实际的执行中，加锁顺序可能会变化导致死锁
    • C++ 提供了 std::lock , 可以一次性锁住多个（两个以上）互斥量，并且没有死锁风险

    • 举个例子，交换操作中使用 std::lock 和 std::lock_guard

      • std::lock 锁住两个互斥量
      • std::adopt_lock 是一个标志，用于告诉 std::lock_guard 或 std::unique_lock，你接手的互斥量已经被上锁，所以在构造时不要尝试加锁，只需要在析构时解锁就可以
    • std::lock 要么将两个互斥量都锁住，要么都不锁
      • 当 std::lock 成功的获取一个互斥量的锁，并且当其尝试从另一个互斥量上再获取锁时，就会有异常抛出，随即第一个锁也随着异常的产生而自动释放

    避免死锁的进阶指导

    虽然锁时产生死锁的一般原因，但是死锁也可能出现在其他地方：例如无锁情况下，每个 std::thread 对象都调用 join()，两个线程就能产生死锁，它们会进入相互等待。下面是一些避免死锁的建议：

    • 避免嵌套锁
      • 一个线程已经获得一个锁时，不要再获取第二个。每个线程只持有一个锁，锁上就不会产生死锁
    • 避免在持有锁时调用用户提供的代码
      • 用户程序可能做任何事，包括获取锁。在持有锁的情况下，用户代码再去获取一个锁，就会违反避免嵌套锁的建议，造成死锁
    • 使用固定顺序获取锁
      • 当硬性条件要求获取两个以上的锁，并且不能使用 std::lock 单独操作来获取它们，那么最好在每个线程上用固定的顺序获取锁
      • 关键是如何在线程之间以一定顺序获取锁
        • 一个线程必须锁住 A 才能获取 B 的锁，锁住 B 才能获取 C 的锁。这将消除死锁发生的可能性
    • 使用锁的层次结构
      • 锁的层次的意义在于提供对运行时约定是否被坚持的检查
      • 需要对应用分层，并且识别在给定层上所有可上锁的互斥量
      • 当代码试图对一个互斥量上锁，在该层锁已被低层持有时，上锁时不允许的
      • 可以在运行时对其进行检查，通过分配层数到每个互斥量上，以及记录被每个线程上锁的互斥量

    • 举个使用层次锁的例子

      • thread_a() 遵守规则，所以运行没问题。thread_a() 调用 high_level_func(),为了获取 high_level_stuff() 的参数对互斥量上锁，让 high_level_mutex 上锁（其层级为 10000），之后调用 low_level_func()，对 low_level_mutex 上锁，这个互斥量有一个低层值 5000；
      • thread_b() 运行就会出问题。首先，它锁住了 other_mutex，这个互斥量的层级值只有 100，这就意味着超低层级的数据已经别保护。当 other_stuff() 调用 high_level_func() 时，就违反了层级结构：high_level_func() 试图获取 high_level_mutex ，这个互斥量的层级值是 10000，要比 100 大得多，因此 hierarchical_mutex 将会产生一个错误，可能会抛出一个异常
    • 当多个互斥量在同一层级时，不能同时持有多个锁。需要每个互斥量在一条链上，并且每个互斥量都比前一个有更低的层级值
    • 同时，在这个例子中可以看到，lock_guard<> 可以与用户定义的互斥量类型一起使用

    简单的层级互斥量实现

    • 每个 hierarchical_mutex 在构造时被分配了一个层次值（hierarchy_value）。值越小，层次越高（优先级越高）；值越大，层次越低。
    • 这个类中使用了 thread_local 的值来代表当前线程的层级值
      • 线程本地变量：允许程序中每个线程都有一个独立的实例拷贝
        • 使用 thread_local 对这样的变量进行声明。
        • 命名空间内的变量、静态成员变量、本地变量都可以声明为线程本地变量
      • this_thread_hierarchy_value 被初始化为最大值，所以最初所有线程都能被锁住
    • 加锁前检查：check_for_hierarchy_violation()
      • 意味着：当前线程已经持有了一个层次值更小（即更高优先级）的锁，若此时再尝试拿一个层次值更大（优先级更低）的锁，就会抛出异常—— 禁止“先低层再高层” 的加锁顺序，强制“高层先行”规则。
    • 当第一个线程锁住一个 hierarchical_mutex 时，this_thread_hierarchy_value 的值是 ULONG_MAX, 这个值会大于其他任何值，所以通过 check_for_hierarchy_violation() 的检查。
    • lock() 代表内部互斥锁已被锁住，就可以更新层级值了
    • 现在锁住另一个 hierarchical_mutex 时，还持有第一个锁，this_thread_hierarchy_value 的值将会显示第一个互斥量的层级值，第二个互斥量的层级值必须小于已经持有的互斥量，检查才能通过
    • 这个类为当前线程存储了之前的层级值，可以调用 unlock() 对层级值进行保存，否则就锁不住任何互斥量
    • try_lock() 与 lock() 功能相似

    std::unique_lock —— 灵活的锁

    std::unique_lock 是一个参数化的互斥量模板类，它提供很多 RAII 类型锁用来管理 std::lock_guard 类型，让代码更加灵活

    • std::unique 不会总与互斥量的数据类型相关
      • 首先，可以将 std::adopt_lock 作为第二个参数传递给构造函数，对互斥量进行管理
      • 也可以将 std::deref_lock 作为第二个参数传递进去，表明互斥量应保持解锁状态
      • 这样就可以被 std::unique_lock 对象的 lock 函数所获取，或传递 std::unique_lock 对象到 std::lock() 中
    • std::unique_lock 会占用比较多的空间，并且比 std::lock_guard 慢一些

    交换操作中 std::unique_lock 与 std::lock() 的使用

    • std::unique_lock 支持 lock(), try_lock(), unlock() 成员函数，所以能将 std::unique_lock 对象传递到 std::lock() 中
    • 这些同名函数仅更新 std::unique_lock 实例中的标志，来确定该实例是否拥有特定的互斥量
      • 这个标志是为了确保 unlock() 在析构函数中被正确调用
        • 如果实例拥有互斥量，那么析构函数必须调用 unlock()
        • 如果实例没有互斥量，析构函数就不能调用 unlock()
      • 这个标志可以通过 owns_lock() 成员进行查询

    不同域中互斥量所有权的转移

    std::unique_lock 没有与自身相关的互斥量，一个互斥量的所有权可以在不同实例中通过移动操作进行转移

    • std::unique_lock 是可移动不可拷贝类型
    • 互斥量所有权转移的一种使用情况是允许一个函数去锁住一个互斥量，并且将所有权转移到调用者上，所以调用者可以在这个锁的保护范围内进行额外的操作

    • 例如：函数 get_lock() 锁住了互斥量，然后准备数据，返回锁的调用函数

      • process_data() 直接转移 std::unique_lock 实例的所有权
    • 通常这种模式会用于已锁的互斥量，其依赖于当前程序的状态，或依赖于返回类型为 std::unique_lock 的函数
    • std::unique_lock 的灵活性也允许实例在销毁前放弃其拥有的锁，对其选择性的释放会提高性能，因为持有锁的时间增加会导致性能的下降

    锁的粒度

    锁的粒度：用来描述一个锁保护着的数据量的大小

    • 一个细粒度锁(a find-grained lock) 能够保护较小的数据量
    • 一个粗粒度锁(a coarse-grained lock) 能够保护较大的数据量‘
    • 锁住互斥量的同时只能对锁内数据进行操作，试图对锁外数据（如对文件输入输出进行上锁）进行处理，多线程带来的性能效益会被抵消

    • std::unique_lock 能在这种情况下正常工作

      • 在调用 unlock() 时，代码不需要再访问共享数据
      • 而后当再次需要对共享数据进行访问时，就可以调用 lock()

    • 锁不仅能锁住合适粒度的数据，还要能控制锁的持有时间，以及什么操作在执行的同时能够拥有锁

    • 对于某些简单的数据类型，拷贝起来很廉价，例如比较两个 int 数据，并且每个被比较的对象都持有该对象的锁。
