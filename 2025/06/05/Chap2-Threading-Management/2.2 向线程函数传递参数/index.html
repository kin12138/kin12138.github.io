    传递指针参数，可能导致悬空指针

    • 向 std::thread 构造函数中的可调用对象或函数传递参数很简单，需要注意：

      • 默认参数要拷贝到线程独立内存中，即使参数是引用形式，也可以在新线程中进行访问

      • 需要特别注意当指向动态变量的指针作为参数传递给线程的情况：

      • 在上述代码中，buffer 是一个 const char * 类型的指针变量，当 oops 函数执行结束就会被销毁

      • buffer 作为参数传递给线程时，类型退化变为 char*, 编译器将 buffer 拷贝到内部存储区，但并不会马上转换为 std::string
        • std::thread 低层在线程启动时才调用 f(3, __), 这里 __ 被用来构造一个临时 std::string。换句话说，临时 std::string 的构造时机在新线程里，而不是 std::thread 对象构造时
        • 如果新线程执行足够“快”，在 oops 函数返回之前就构造了 std::string ，此时 buffer 还在栈上有效，暂时不会出错
        • 如果新线程的执行稍慢，刚好在 oops 执行结束，释放 buffer 指向的内存之后启动，后续线程再去读取这块已经被释放的内存就会导致访问野指针，造成未定义行为

    显示的类型转换避免传递悬空指针或引用

    • 解决方案就是在 std::thread 构造函数之前就将 buffer 转换为 std::string 类型

      • 这种方法将 buffer 在传递给 std::thread 构造函数之前转换为 std::string, 这样 std::thread 构造函数将这个 std::string 对象进行一份拷贝，使得这份数据在分离线程中得以独立
      • 这时，数据经过 std::thread 构造函数拷贝之后，与原始线程中的数据分离开，在新线程中对数据的任何更改都无法影响原始线程中的数据

    传递引用

    • 而有时，程序还可能需要原始线程中的数据与新线程中的数据同步，这时可以传递一个引用

      • update_data_for_widget 的第二个参数期望传递一个引用，但是 2 处 std::thread 的构造函数并不知晓，它盲目的拷贝已经提供的变量，当线程调用 update_data_for_widget 函数时，传递给函数的参数是 data 变量内部拷贝的引用，而非数据本身的引用，当线程结束时，内部拷贝数据将会在数据更新阶段被销毁， process_widget_data 将接收到没有被修改的 data

      • 解决这个问题方法就是给 std::thread 构造函数传递 data 的引用

      • 在进行了上述改变之后，update_data_for_widget 将直接接收到 data 变量的引用，当线程结束之后，process_widget_data 也将收到经过子线程内部改变后的 data

    std::thread 和 std::unique_ptr 相似

    • std::thread 不会像 std::unique_ptr 一样独占对象所有权，但是它会占用一部分资源的所有权：每个实例都管理一个执行线程
    • std::thread 所有权可以在多个实例中相互转移，因为这些实例都是 moveable and aren’t copyable
