    线程管理基础

    • 每个线程至少有一个线程：执行 main() 函数的线程， 其余线程各自有其入口函数

    启动线程

    • 线程在 std::thread 对象创建时启动

    • std::thread 可以接受可调用对象作为构造参数

      • 接受一个重载了调用运算符的类类型，需要避免语法解析问题

      • 这里会被解析为：声明一个名为 my_thread 的函数，带有一个参数（函数指针指向没有参数并且返回 background_task 对象的函数），返回一个 std::thread 对象

      • 避免这个问题的方法有三种：

        • 多括号
        • 统一的列表初始化方式
        • lambda 表达式

    • 启动线程之后，需要明确是要等待线程结束，还是让其自主运行（join 或 detach）

      • 如果在 std::thread 对象销毁之前还没有做出决定，程序就会调用 std::terminate() 终止

    • 如果不等待线程，就必须保证线程结束之前，可访问数据的有效性

      • 对象销毁之后再去访问，将产生未定义行为

      • 例如：函数已经结束，线程依旧访问局部变量

      • 处理这种问题的常规方法是：使得线程功能齐全，将数据拷贝到线程中，而并非拷贝到共享数据（主线程栈空间）中

    等待线程完成(join)

    • 对于上面例子中的问题，可以使用 join 等待线程完成，就可以确保局部变量在线程完成后，才被销毁
    • 在上面例子中，原始线程什么都没做，但是实际编程中，原始线程要么有自己的工作，要么开启多个子线程进行一些工作，然后等待这些线程的结束
    • join() 是简单粗暴的等待线程完成或者不等待，如果想要更灵活的处理，需要其他机制来完成，如条件变量、futures
    • 调用 join() 的行为，还清理了线程相关的存储部分。
      • 意味着只能对一个线程使用一次 join ；一旦使用过 join(), std::thread 对象将不能再次使用 join()，当对其调用 joinable() 时，返回 false

    特殊情况下的等待

    • 如果线程运行之后发生异常，并在 join 调用之前抛出，就意味着这次调用会被跳过

    • 通常，为避免程序被异常所终止需要在异常处理过程中调用 join(), 从而避免生命周期问题

    • 还有一种方式是使用 RAII(Resource Acquisition Is Initialization 资源获取及初始化) 方式，并且提供一个类，并在析构函数中使用 join()

      • 执行到 4 处时，局部对象就要被逆序销毁， thread_guard 对象 g 是第一个被销毁的，这时线程在析构函数中被 join 到原始线程，即使原始线程中的 do something in current threading 抛出异常，这个销毁过程依旧存在，抛出异常不会影响线程的执行
      • 在 1 处，必须判断线程是否是 joined ，一个线程只能调用一次 join()
      • 在 3 处，将拷贝构造函数与拷贝赋值运算符定义为 delete ，防止编译器自动生成，因为对 std::thread 的拷贝或赋值是十分危险的，这可能会弄丢已经调用 join() 的线程

    后台运行线程(detach)

    • 使用 detach() 让线程进入后台运行，主线程不能与之直接交互
    • 通常称分离线程为守护线程(daemon threads)
    • 不能对没有执行线程的 std::thread 对象使用 detach() 或 join(), 若 std::thread 对象调用 joinable() 的结果是 true ，则可以调用 detach() ，join() 同理

    • 例子：使用分离线程处理其他文档

      • 如果用户选择打开一个新文档，为了让其迅速打开，需要启动一个新线程去打开新文档，并分离线程。
      • 与当前线程所做工作一样，新线程不过是新打开了一个文档，所以 edit_document 函数可以复用，通过传参形式打开文件
      • 上述例子也介绍了传参启动线程的方法，不仅可以向 std::thread 构造函数传递函数名(可调用对象), 还可以传递函数所需实参

    std::thread 的关键接口

    Interface Description
    std::thread::joinable() 检查该 std::thread 对象是否持有一个可被 .join() 或 .detach() 的线程
    std::thread::join() 阻塞当前线程直到调用线程函数结束，清理资源
    std::thread:::detach() 让目标线程脱离当前 std::thread 对象管理， 成为守护线程，后续无法再 join
    std::this_thread::get_id() 返回当前线程的唯一 ID
    std::this_thread::sleep_for(...)
    std::this_thread::yield()    		 用于让当前线程休眠一段时间或主动让出时间片

    • fib_range 函数负责计算区间内的斐波那契数，参数 start 是开始位置，count 表示要连续计算多少个数
    • 因为所有的线程同时打印到标准输出，因此打印顺序会交错

    join() VS detach()

    • 什么时候用 join
      • 如果子线程执行的任务必须在主线程或另一个线程做出下一步动作之前完成，必须使用 join 来同步
      • 典型场景：在程序退出前需要确保所有子线程已经完成自己的工作，防止数据不一致，资源泄漏
    • 什么时候用 detach
      • 子线程的工作并不影响主线程的后续逻辑，且不用等它结束，只希望它在后台运行
      • 典型场景：记录日志、后台监控、定时任务等，既希望一启动就 Running， 又不希望堵住主线程
    • 注意: 如果使用 detach，必须确保:
      • 子线程访问的任何数据在子线程整个生命周期中始终有效
      • 用户不需要关心它的结束状态，也不需要获取它的返回值

    小项目：启动守护线程执行后台日志

    • stop_flag 为 std::atomic<bool>, 在多线程场景下保证可见性
      • std::atomic<bool> 提供了原子操作，确保所有线程访问时的读写操作是原子性的，不会发生竞态。它还能保证对该变量的修改对所有线程立即可见，从而确保线程之间正确同步。
    • 主线程在检测到 quit 后等待两秒，让守护线程打印完退出信息
    • 当主线程返回 main() 并结束时，整个进程都会终止，包括所有的守护线程。操作系统会在进程退出时回收所有线程资源，因此守护线程无法在进程结束后继续运行

    Questions And Answer

    问题 1：C++ 中线程的生命周期如何定义？

    • 回答：线程生命周期包括创建、运行、终止三个阶段。创建使用 std::thread，运行时执行传入函数，终止后需 join() 或 detach() 回收资源；否则会调用 std::terminate()。

    问题 2：如何使用 std::thread 创建线程并输出线程 ID？

    • 回答：见示例 1，使用 std::thread t(hello); 创建线程，hello() 内调用 std::this_thread::get_id() 获取 ID，主线程调用 t.join() 等待结束。

    问题 3：如何启动多个线程并分别计算不同区间的斐波那契数？

    • 回答：见示例 2，用 std::vector<std::thread> 存储多个 std::thread 对象，每个线程传入不同 start 值，最后遍历 join。

    问题 4：std::thread 对象析构时，若未 join() 或 detach() 会发生什么？

    • 回答：如果 std::thread 管理的线程仍可 joinable() 且未调用 join() 或 detach()，在析构时会调用 std::terminate()，导致程序异常终止。

    问题 5：为什么用 std::atomic<bool> 而不是普通 bool 作为停止信号？

    • 回答：普通的 bool 在多线程写后，其他线程不一定能立即看到变化，会发生数据竞争。std::atomic<bool> 保证读写原子性，并让修改对所有线程立即可见，从而正确同步。

    问题 6：守护线程在主线程退出后还能否继续运行？

    • 回答：当主线程结束并返回 main() 时，整个进程会终止，包括所有守护线程。操作系统在进程退出时回收所有线程资源，守护线程无法在进程已终止后继续运行。

    问题 7：在使用 detach() 分离线程后，如何判断该线程是否已经结束？

    • 回答：分离后，std::thread 对象不再管理实际线程，无法直接获取其状态。必须自行设计状态通告机制，例如：

      1. 使用共享的 std::atomic<bool> 标志，在子线程结束前将其置为 true，外部循环检查该标志；
      2. 使用 std::condition_variable 和 std::mutex，在线程结束时通知观察者；
      3. 使用 std::promise/std::future，让子线程在结束时设置 promise，外部通过 future.wait_for 或 future.wait 判断。
