    除了互斥量之外，还有很多其他保护共享数据的方式，提供更加合适的保护

    • 例如在任何情况下，数据初始化后锁住一个互斥量，纯粹是为了保护其初始化过程，这是降低性能

    保护共享数据的初始化过程

    延迟初始化(Lazy initialization) 在单线程代码中很常见，每个操作都需要先对源进行检查，为了了解数据是否被初始化，然后在其使用前决定数据是否需要初始化

    为了使共享数据对并发访问时安全的，下面的代码会使线程资源产生不必要的序列化，这是因为每个线程都要等待互斥量，从而去确定数据源已经被初始化了

    • 在多线程中，如果不对共享数据加以保护，就会发生竞态 —— 多个线程同时检测到尚未初始化，而导致：

      • 重复初始化
      • 部分线程只看到半初始化状态

    • 使用一个互斥量的延迟初始化（线程安全）过程

    典型错误做法：双重检查锁(Double-Checked Locking)

    • 问题：CPU 和编译器重排序会使 “读取 resource_ptr” 和 some_resource 构造完成顺序错乱

    std::once_flag 和 std::call_once

    C++ 提供了 std::once_flag 和 std::call_once 来处理初始化时产生竞态的情况

    • 每个线程只需要使用 std::call_once, 在 std::call_once 结束时，就能安全的知道指针已经被其他的线程初始化了
    • std::call_once 比显式的使用互斥量消耗资源更少
    • std::call_once 可以和任何函数或可调用对象一起使用

      • 之后其他线程再次调用 std::call_once 会立即返回，不会再执行 open_connection()

    static 局部变量的初始化存在条件竞争

    将一个局部变量声明为 static，变量在声明后就已经完成初始化。对于多线程调用的函数，这里就有条件竞争：抢着去定义这个变量

    • 解决方式：使用一个函数去定义和初始化这个 static 变量，然后在线程调用函数中用 std::call_once 调用这个初始化 static 变量的函数

    保护很少更新的数据结构

    某些情况下，数据在很长一段时间内只需要让多线程去访问，只需要在极少的定期时间去更新它的数据，这种情况下，使用 std::mutex 使得在没有发生修改时，将削减并发读取数据的可能性

    • 这种情况下，可以使用 Boost 库提供的 “读者-写者锁”(reader-wirter mutex)
      • 一个“写者”线程独占访问和共享访问，多个“读者”线程并发访问
    • 使用 std::lock_guard<boost::shared_mutex> 和 std::unique_lock<boost::shared_mutex> 进行上锁，作为 std::mutex 的替代方案，这样就保证更新线程的独占访问
    • 其他更新线程不需要修改数据结构，可以使用 boost::shared_lock<boost::shared_mutex> 获取共享访问权
    • 例子：一个 DNS 缓存，使用 std::map 持有缓存数据，使用 boostd::shared_mutex 进行保护
    • 在 update_or_add_entry() 函数调用时，独占锁会组织其他线程对数据结构进行更改，并且这些线程在这时也不能调用 find_entry()
    • find_entry() 使用了 boost::shared_lock<> 实例来保护其共享和只读权限，多线程可以同时调用 find_entry()，且不会出错

    嵌套锁

    • 一般，当一个线程已经获取一个 std::mutex 实例时（已经上锁），并再次对其上锁，这个操作时错误的，并且继续尝试这样做的话，会产生未定义行为
    • 然而在某些情况下，一个线程尝试获取一个互斥量多次，而没有对其进行一次释放，这种行为是可行的，C++ 中提供了这种互斥量 std::recursive_mutex （递归锁）
    • std::recursive_mutex 功能上与 std::mutex 类似，除了它可以让线程在单个实例上获取多个锁。但是注意：对这个互斥量调用几次 lock() ，就必须也调用几次 unlock()
    • 嵌套锁一般用在可被多线程并发访问的类上，其拥有一个互斥量保护其数据
      • 每个 public 成员函数都会对互斥量进行上锁，然后完成自己的工作，之后再解锁
      • 不过当一个 成员函数的工作需要调用另一个成员函数时，第一个成员函数已经对互斥量上过锁了，第二个成员函数还会尝试对其上锁，这就会产生未定义行为
      • 这种情况下将互斥量转换为嵌套锁将解决这种问题
