    使用消息传递的同步操作

    CSP(Communicating Sequential Processor), 通讯顺序进程：

    • 当没有共享数据，每个线程可以进行独立思考，其行为纯粹基于所接受到的信息
    • 每个线程都有一个状态机：当线程收到一条信息，它将会以某种方式更新其状态，并且可能向其他线程发出一条或多条信息，对于消息的处理依赖于线程的初始化状态
    • 这样，无论采用什么方式去实现每个线程，任务都会分成独立的处理部分，这样会消除潜在的混乱（数据共享并发）

    ATM 示例笔记

    1. 项目概述

    • 基于消息传递框架的多线程自动取款机 (ATM) 模拟示例
    • 展示了 Actor 模型、线程间通信、状态机、类型分发和资源管理等核心概念

    2. 消息传递框架

    2.1 核心组件

    • message_base: 空基类，所有消息的基类
    • wrapped_message: 模板类，封装任意消息类型，实现类型擦除

    • queue: 封装 std::queue<shared_ptr<message_base>>、mutex、condition_variable

      • push(msg): 将消息入队并调用 notify_all()
      • wait_and_pop(): 阻塞等待队列非空或关闭，弹出并返回消息，或在关闭时抛出 close_queue
    • sender: 持有 queue 的 shared_ptr，提供 send(msg) 方法给外部线程发消息
    • receiver: 拥有 queue，提供 wait() 方法，得到一个可链式调用的调度器

    2.2 调度器(Dispatcher)

    • dispatcher: 最底层，负责检测 close_queue 并在析构时循环 wait_and_dispatch()

    • TemplateDispatcher\:

      • 持有前一个调度器 prev 和回调 f（function<void(Msg)>）
      • handle<NextMsg>(f_next): 返回新的 TemplateDispatcher<this, NextMsg>，实现链式调用
      • dispatch(msg): dynamic_cast 检查 Msg 类型，匹配则执行回调并返回 true；否则 return prev->dispatch(msg)
      • 析构时调用 wait_and_dispatch()，循环处理直到第一个 handler 匹配或遇到 close_queue

    3. ATM 示例架构

    3.1 主要角色 (Actors)

    • UI (interface_machine): 用户界面线程，打印提示和操作结果

      • 消息队列: UI Queue
      • 消息类型: card_inserted, issue_money, display_balance, display_insufficient_funds, eject_card, close_queue

    • ATM (atm): ATM 状态机线程，核心流程控制

      • 消息队列: ATM Queue
      • 消息类型: card_inserted, digit_pressed, withdraw_pressed, verify_pin, withdraw, get_balance, pin_verified, withdraw_ok, withdraw_denied, balance, eject_card, cancel, close_queue

    • Bank (bank_machine): 银行后台线程，处理账户逻辑

      • 消息队列: Bank Queue
      • 消息类型: verify_pin, withdraw, get_balance, pin_verified, pin_incorrect, withdraw_ok, withdraw_denied, balance, close_queue

    3.2 线程通信流程

    1. UI 发 card_inserted 到 ATM Queue
    2. ATM 在 state_wait_for_card 中接收，切换到 state_get_pin
    3. ATM 收集 PIN，发 verify_pin{account,pin,atm_queue} 到 Bank Queue
    4. Bank 接收，处理后发回 pin_verified 或 pin_incorrect 到 ATM Queue
    5. ATM 接收后，根据类型切换到操作等待或退卡状态
    6. ATM 发 withdraw 或 get_balance 到 Bank Queue，Bank 返回结果，ATM 再发显示消息到 UI Queue
    7. UI 接收并打印，最后 ATM 发 close_queue 优雅关闭所有线程

    4. 关键技术与学习点

    • Actor 模型: 独立角色、私有队列、点对点通信
    • 线程安全队列: mutex + condition_variable 实现生产者-消费者
    • 类型擦除与 RTTI 分发: dynamic_cast 实现安全类型匹配
    • 链式 API 设计: 每次 handle<>() 返回新调度器，临时对象析构启动循环
    • 有限状态机建模: 用函数指针或 std::function 管理状态切换
    • 优雅退出: 通过 close_queue 消息和异常清理循环
    • 资源管理: shared_ptr 管理消息生命周期，避免悬空和泄漏

    5. 推荐练习

    1. 新增消息类型: 如 cancel_transaction，并在 ATM 中处理
    2. 实现超时: 在 wait_and_pop() 加超时版本，模拟用户超时退卡
    3. 日志 actor: 添加一个日志线程，记录所有消息及时间戳
    4. 无锁队列尝试: 用 lock-free 结构替换 queue，对比性能
    5. 可视化: 使用 Excalidraw 或 Mermaid 绘制完整消息流图

    6. 用户提问与回答

    以下是用户在本示例中提出的问题及对应简要回答：

    1. interface_machine 中 std::mutex iom 定义及作用

      • 定义位置：在 interface_machine 类的私有成员中，与 incoming（receiver）同级声明。
      • 作用：在多线程并行写入同一 std::cout 时，通过 std::lock_guard<std::mutex> lk(iom); 保护输出，防止不同消息处理分支的打印内容交错。

    2. 终极守门员 dispatch 函数处理 close_queue 的机制及异常终止循环

      • 动态识别：使用 dynamic_cast<wrapped_message<close_queue>*> 检查是否为关闭信号。
      • 异常抛出：识别到后 throw close_queue();，让最外层的 wait_and_dispatch() 捕获并跳出循环，实现优雅退出。

    3. dispatch 返回 false 后如何沿链条上传并在循环中表现

      • 链式调用时，每层 dispatch 未处理时都会执行 return prev->dispatch(msg);，将消息交给上一层调度器。
      • 在模板 wait_and_dispatch 中，只有当 dispatch(msg) 返回 true（消息被处理）或抛出 close_queue，循环才终止；返回 false 则继续下一次循环，等待新消息。

    4. 链式调用底层实现机制： handle 如何构造并链接调度器

      • 每次调用 .handle<Msg>(f) 都返回一个新的临时 TemplateDispatcher<Prev, Msg> 对象，其中 prev 指向前一个调度器，f 为回调。
      • 通过临时对象的构造顺序（左到右）与析构顺序（右到左），最外层临时析构时启动唯一一次的消息循环。

    5. 银行与用户界面状态机流程图

      • 可使用 Mermaid 或 Excalidraw 绘制两条状态机图，分别展示状态和事件转换。
      • 之前已提供 Mermaid 代码示例，并可导入 Excalidraw JSON 进行可视化编辑。

    6. queue 的 push 与 wait_and_pop 中锁和 notify_all 的必要性

      • 即使队列属于单个 actor，由于其他线程（生产者）会并发调用 push，消费者调用 wait_and_pop，仍需 mutex 保护容器内部结构，防止数据竞争。
      • condition_variable 必须搭配 mutex 使用，notify_all 唤醒所有等待线程后通过锁竞争决定哪个线程先拿到消息。

    7. 消息队列点对点路由设计，避免消息误投与丢失

      • 每个 actor 拥有独立的 queue，sender.send() 明确指定目标队列，只有目标线程才能从该队列消费消息，绝无误投可能。

    8. 初学者从 ATM 示例中可以学习到的关键知识点

      • Actor 模型、线程安全队列、类型分发、状态机建模、链式 API 设计、异常驱动退出、资源管理等。

    9. 用户界面线程通过 sender 向 atm_thread 发送消息的流程

      • UI 线程持有通过 atm.get_sender() 获取的 sender，调用 send(msg) 将消息封装为 wrapped_message 并入 atm_queue，由 ATM 线程在 incoming.wait() 循环中消费。
